# 다양한 연관관계 매핑
## 목차
* 연관관계 매핑시 고려사항 3가지
  

* 다중성
    * [다대일(N:1)](#다대일)
    * [일대다(1:N)](#일대다)
    * [일대일(1:1)](#일대일)
    * [다대다(N:N)](#다대다)
***

## 연관관계 매핑시 고려사항 3가지


## 다대일
* `다`가 연관관계 주인인 경우를 뜻함.

## 일대다
* `일`이 연관관계 주인인 경우를 뜻함.
* `DB입장`에서는 무조건 `다`에서 외래키가 들어가야하기 때문에 반대편 테이블의 외래 키를 관리하는 특이한 구조.
* `@JoinColumn`을 꼭 사용해야 한다. 미기입 시 중간 테이블이 생기는 조인테이블 방식을 사용함.
* 예시 
  * `Team.class` (`Member.class`에서는 Team을 아예 지우면 된다.)
    ```java
    public class Team {
        
        @Id @GeneratedValue
        @Column(name = "team_id")
        private Long id;
        
        private String name;
    
        @OneToMany
        @JoinColumn(name = "team_id")
        private List<Member> members = new ArrayList<>();
    }
    ```
  * 테스트 `OneToManyCase.class`
    ```java
    public class OneToManyCase {
        public static void main(String[] args) {
    
            EntityManagerFactory emf = Persistence.createEntityManagerFactory("hello");
            EntityManager em = emf.createEntityManager();
    
            EntityTransaction transaction = em.getTransaction();
            transaction.begin();
            try {
                Member member = new Member();
                member.setUserName("onetomanyMember1");
                //멤버 저장
                em.persist(member);
    
                Team team = new Team();
                team.setName("onetomanyTeam1");
                // 외래키가 MEMBER 테이블에 있기때문에 결국 Member테이블을 건드림.
                team.getMembers().add(member);
                // 외래키 업데이트
                em.persist(team);
    
                transaction.commit();
            } catch (Exception e){
                transaction.rollback();
            }
            finally {
                em.close();
            }
            emf.close();
        }
    }
    ```
* 개발을 할 때 혼동이 올 수 있음.
* `일대다 양방향 정리`
  * 공식적으로 존재하지 않음.
  * `@ManyToOne과 @JoinColumn(insertable=false, updatable=false)`를 사용해서 읽기 전용 필드로 만들어 양방향 처럼 사용.
  * 왠만하면 `다대일 양방향`을 사용하자. (한번씩 쓰일때도 있음.)
  
## 일대일
 * 주 테이블이나 대상 테이블 중 외래키 선택 가능
   * `주 테이블`에 외래 키
     * 주 객체가 대상 객체의 참조를 가지는 것 처럼, 주 테이블에 외래키를 두고 대상 테이블을 찾아 혼동이 없다.  
       => 주 테이블만 조회해도 대상 테이블에 데이터가 있는지 확인이 가능하다.
     * JPA 매핑이 편리하다.
     * 값이 없으면 외래 키에 `Null`을 허용해야 한다.
    
   * `대상 테이블`에 외래 키
     * 주 테이블과 대상 테이블을 **일대일에서 일대다 관계로 변경할 때 테이블 구조 유지 가능.**
     * 프록시 기능의 한계로 **지연 로딩으로 설정해도 항상 즉시 로딩됨.**
  
* 외래 키에 데이터베이스 `유니크 제약조건`을 추가해야함.
* `다대일` 연관관계 구조와 같다.

* 고민해야 할 점.
  * DB : 테이블은 한번 만들어지면 변경하기가 어려움. 추후 **비즈니스 로직이 바뀌는 것을 고려해 어느것을 외래키로 사용할지 고민이 필요.**
 * 개발 : 개발단계에서 성능을 고려해서 또 어느것을 외래키로 사용할지 고민  
   예시 : `Member`, `Locker`을 고려했을때 검색 빈도는 `Member`가 압도적으로 높으므로 `Member`를 외래키로 사용하는 것이 유리함.
  
## 다대다
사실 다대다는 **실무에서 사용하면 안된다.**  
 * 테이블
   * 관계형 DB는 정규화된 테이블 2개로 다대다 관계를 표현할 수 없음.
   * **연결 테이블을 추가**해서 `일대다`, `다대일` 관계로 풀어내야 함. (중간 조인 테이블이 필요하다.)
  
 * 객체
   * 객체는 컬렉션을 사용해서 다대다 관계가 가능함. (`@ManyToMany`, `@JoinTable`)

 * 문제점
   * 여기서 중간 조인 테이블은 연결하는 기능만 수행한다. **하지만, 연결 테이블은 단순히 연결만 하고 끝나지 않는다.**
   * `중간 조인 테이블`에는 매핑정보만 들어가고 추가정보는 입력하지 못한다.

 * 극복방안
   * 연결 테이블용 엔티티 추가(중간 연결 테이블을 `엔티티로 승격`)
   * `@ManyToMany`를 `@OneToMany` + `@ManyToOne`으로 표현 가능.
